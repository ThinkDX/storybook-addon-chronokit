# The `date` parameter

Everything chronokit does is controlled by one story parameter: `date`. Set it on
a story, a component's `meta`, or globally in `.storybook/preview`.

## Shorthand

Pass a date directly to freeze the clock at that instant:

```ts
parameters: { date: '2025-06-01T12:00:00' } // string, number (ms), or Date
```

## Full form

```ts
parameters: {
  date: {
    now: new Date('2025-06-01T12:00:00').getTime(),
    canProgress: true,
    clockSpeed: 30,
  },
}
```

| Field         | Type                                | Default | Description                                                                 |
| ------------- | ----------------------------------- | ------- | --------------------------------------------------------------------------- |
| `now`         | `string \| number \| Date`          | —       | The mocked "current" time. Required.                                        |
| `canProgress` | `boolean \| ((context) => boolean)` | `false` | `false` freezes the clock at `now`; `true` lets it advance. May be a predicate. |
| `clockSpeed`  | `number`                            | `1`     | Real-time multiplier while progressing — e.g. `30` makes time run 30× faster. |

### Recommended setup: static `now` + progress on interactions

`canProgress` can be a function that receives the story context and returns a
boolean, so you can set the rule **once globally** instead of per story. The
recommended default: pin a single canonical `now` (freezing the clock) and let it
run only for stories that have a play function.

```ts
// .storybook/preview.ts
const STORYBOOK_NOW = new Date('2025-06-01T12:00:00').getTime()

export default {
  decorators: [mockDateDecorator],
  parameters: {
    date: {
      now: STORYBOOK_NOW,
      canProgress: (ctx) => !!ctx.playFunction,
    },
  },
}
```

Why this is a good default:

- **Deterministic snapshots** — every story renders at the same frozen instant,
  so visual-regression tools (e.g. Chromatic) don't flap on dates, relative
  timestamps, or "today" highlights.
- **Interactions still work** — play functions usually rely on waits and timers,
  so the predicate hands *them* a live clock automatically. No per-story
  `canProgress: true`.

Any story can still override `now` / `canProgress` / `clockSpeed`.

## How it behaves

- **Frozen** (`canProgress: false`) — `Date.now()`, `new Date()`, timers and
  animation frames all see a clock pinned at `now`. The UI never changes on its
  own. See **Static Time**.
- **Progressing** (`canProgress: true`) — the clock advances from `now`. With
  `clockSpeed` it advances faster than wall time. See **Controlled Dynamic Time**.

The decorator restores the real clock when the story unmounts, so stories stay
isolated from one another.
